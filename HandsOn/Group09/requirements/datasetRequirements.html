Selected datasets

  1. City of Melbourne — Pedestrian Counting System (Monthly counts per hour)

    Identifier: pedestrian-counting-system-monthly-counts-per-hour

    Updated: monthly

    Theme: Transport

    Language: English

    Key columns: id, location_id (aka sensor_id), sensing_date (date), hourday (hour), direction_1, direction_2, pedestriancount (hourly total), sensor_name, location (geo point).

    Hourly pedestrian counts since 2009; zeros appear when no pedestrians passed. It can be merged with “Sensor Locations” via sensor_id/location_id.
  2. City of Melbourne — Pedestrian Counting System · Sensor Locations

    Identifier: pedestrian-counting-system-sensor-locations

    Updated: daily

    Theme: Transport

    Language: English

    Key columns: location_id, sensor_name, sensor_description, status (A/I/R), installation_date, note, latitude, longitude, location (geo point).

    Records bi-directional pedestrian movement; notes document moved/removed/inactive sensors. Join to related datasets via sensor_id/location_id.

Requirements assessment (R1–R6)

Req. What it asks Status Evidence / comments
R1 Smart-city domain Meets Mobility / use of public urban space. Hourly pedestrian activity across city locations supports planning, safety and events/tourism analysis. Both datasets are under the Transport theme and describe city sensors and their counts.
R2 Available as CSV Meets Official portal provides direct CSV export for both datasets (and API). A historical CSV attachment is provided for counts.
R3 Open license Meets Both datasets are published under Creative Commons Attribution 4.0 (CC BY 4.0). License text: https://creativecommons.org/licenses/by/4.0/legalcode.
R4 Linkable to generic entities Meets Each hourly row carries sensor_id/location_id → 1-to-1 join with “Sensor Locations” to attach coordinates, status and notes. Places (coordinates / names) can be aligned to generic entities (e.g., Wikidata POIs, districts; public holidays as events).
R5 (optional) Documentation Meets Resource pages include a Data Dictionary / schema with field names and types (e.g., sensing_date, hourday, pedestriancount).
R6 (optional) Multiple sources Meets We link two official sources from the same portal: Monthly counts per hour + Sensor Locations. (Optional later: enrich with public holidays/events or weather.)

Entities in the selected datasets that can be linked

  • Locations: latitude/longitude, location name → link to POIs/districts.
  • Sensors: sensor_id/location_id, sensor_name, status, installation_date, note.
  • Time: sensing_date + hourday (hour) → link to public holidays/events.

Entities in other datasets that can be linked to ours

  • Wikidata / DBpedia: POIs (squares, stations, malls), administrative areas (districts), and Australian public holidays.
  • Geonames: toponyms and administrative hierarchy.
  • Open weather (optional): daily/hourly summaries for correlation analysis.

Scope note: This document only covers dataset selection and discussion of R1–R6 (R5–R6 optional). No servers or endpoints are assumed at this stage.
